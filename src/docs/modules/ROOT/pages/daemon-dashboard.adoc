= Sluice Daemon Dashboard
:description: Running the jdbc-export daemon subcommand and using the Sluice browser operations console.

The `daemon` subcommand keeps `jdbc-export` alive as a long-running process and serves *Sluice* — an HTMX-powered browser operations console — at `http://<host>:<port>/`.

Sluice provides a form-driven interface for submitting export jobs, monitoring live runs, and managing saved database connections, all without leaving the browser.

NOTE: Job history is held in memory only and is lost when the daemon stops. Saved connections do persist across restarts — see xref:connections.adoc[].

== Starting the daemon

[source,bash]
----
java -jar build/quarkus-app/quarkus-run.jar daemon [--port 8080] [--host localhost] [--allow-remote]
----

The process blocks until interrupted (Ctrl-C or SIGTERM). Once running, open `http://<host>:<port>/` in a browser to access Sluice.

.Daemon options
[cols="1,1,3",options="header"]
|===
|Option |Default |Description

|`--port`
|`8080`
|TCP port the HTTP server binds to.

|`--host`
|`localhost`
|Interface address to bind. Must be a loopback address unless `--allow-remote` is also passed.

|`--allow-remote`
|_(flag, off)_
|Permits binding to a non-loopback address. See xref:security.adoc[] before enabling.
|===

IMPORTANT: Running the packaged jar with no arguments is a usage error — `jdbc-export` expects `--url` for a one-shot export. The `daemon` subcommand must be requested explicitly.

=== Development mode

[source,bash]
----
./gradlew quarkusDev
----

Running `quarkusDev` with no additional arguments boots the daemon with hot reload enabled. The Quarkus Dev UI is available at `/q/dev-ui`. Changes to templates and source files are reflected immediately without restarting the process.

== Sluice: the browser operations console

Sluice uses a *Kinetic Green* colour theme with a dark/light toggle. The selected theme and the active view both persist across page reloads. A left side-rail provides navigation between the four views described below.

No authentication is required. Security is enforced through a loopback-only bind address and a mandatory CSRF request header. See xref:security.adoc[] for the full security model.

== Dashboard view

The Dashboard view is the primary entry point for submitting export jobs.

=== KPI strip

Above the submission form, a row of key performance indicators shows live daemon statistics:

* Daemon uptime
* Total jobs submitted
* Total rows exported
* Active runs
* Success rate

=== New export form

The form accepts the following fields:

.Export form fields
[cols="1,3",options="header"]
|===
|Field |Description

|Connection picker
|Optional. Select a saved connection to auto-fill the JDBC URL, user, and password-env fields.

|JDBC URL
|Full JDBC connection URL (e.g. `jdbc:postgresql://localhost:5432/mydb`).

|User
|Database username.

|Password
|Database password entered directly.

|Password env
|Name of an environment variable whose value is the database password. Mutually exclusive with the password field.

|SQL
|The read-only SQL query to execute (`SELECT` or `WITH` only).

|Output path
|File system path for the exported output file.

|Format
|Segmented control: `csv` / `tsv` / `json` / `ndjson` / `parquet`.

|Parquet codec
|Compression codec selector, shown when the `parquet` format is selected.

|Overwrite if exists
|Toggle. When enabled, an existing output file at the specified path will be replaced.
|===

The *Describe schema* button introspects the query and returns the column names and JDBC types without streaming any data rows.

The *Run export* button submits the job. Submitted jobs appear immediately in the running mini-card or the up-next queue below the form.

=== Running and queued jobs

Directly below the form, two compact panels show:

* *Running* — a mini-card for the currently executing job.
* *Up next* — the queue of jobs waiting to start.

=== Job queue and history table

A filterable table lists all jobs, past and present. Filter chips narrow the view:

`All` | `Running` | `Queued` | `Completed` | `Failed`

Each row shows the job status and format as pills, the row count, and the job duration.

== Live Run view

The Live Run view is a metrics cockpit for the currently executing job. It refreshes approximately every 1.5 seconds.

The following live metrics are displayed:

.Live Run metrics
[cols="1,3",options="header"]
|===
|Metric |Description

|Rows streamed
|Running count of rows written to the output file.

|Throughput
|Live chart of rows per second over the duration of the run.

|Elapsed time
|Wall-clock time since the job started.

|Output size
|Size of the growing output file on disk.

|Column count
|Number of columns returned by the query.

|Fetch-batch progress
|JDBC fetch size and batch progress indicator.

|JVM heap gauge
|Current JVM heap usage and capacity.

|Connection and query state
|Database server, version, driver name, and current JDBC connection state.

|Live log
|Synthesised, time-stamped log entries for the active job.

|Running SQL
|The SQL statement being executed, alongside writer configuration details (format, codec, output path).
|===

TIP: The live metrics are sourced directly from the running exporter — row counts are reported as the `ResultSet` is streamed, and the output file size is read from disk in real time.

When no job is running, the view shows an empty state and a summary card for the most recently completed run.

== Schedules view

The Schedules view is fully functional and provides complete management of export schedules directly from the dashboard.

The operator can perform the following actions:

* *List* all defined schedules with their current status and next scheduled run.
* *Create* a new schedule by specifying the SQL query, connection, output format, and trigger type.
* *Edit* an existing schedule to update any of its parameters.
* *Delete* a schedule that is no longer required.
* *Enable or disable* a schedule without removing it.
* *Run now* to trigger an immediate one-off execution of any schedule, independent of its next scheduled time.

Both recurring and one-off schedules are supported. Recurring schedules use a cron expression; one-off schedules fire once at a specified date and time and are then automatically disabled.

When entering a cron expression, a live cron-to-English hint is displayed directly beneath the field, translating the expression into a human-readable description as you type.

An upcoming-runs timeline is shown for each recurring schedule, listing the next N scheduled execution times so the operator can verify the cadence before saving.

Schedules are persisted to `~/.sluice/schedule.json` and are fired by the built-in scheduler running inside the daemon process. No external scheduler or cron daemon is required.

See xref:schedules.adoc[] for the schedule format, trigger conditions, and REST API.

== Connections view

The Connections view provides basic management of saved JDBC connection profiles:

* *Add* a new named connection with a JDBC URL, user, and optional password-env reference.
* *Edit* an existing connection profile.
* *Delete* a saved connection.
* *Test* a connection to verify it is reachable before submitting an export.

Connections selected in the Dashboard view auto-fill the corresponding form fields.

See xref:connections.adoc[] for full detail on connection storage, file format, and configuration.
