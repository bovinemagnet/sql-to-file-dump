= Connection Registry

The daemon maintains an operator-managed registry of reusable JDBC connections.
Saved connections let you define database credentials once and reference them across multiple export jobs from the xref:daemon-dashboard.adoc[Dashboard], without repeating connection details or exposing passwords in the browser.

== Overview

When the daemon is running, the connection registry provides a central store of named JDBC connections.
Each connection captures the URL, username, and the *name* of an environment variable that holds the password — never the password value itself.

Connections are persisted to disk between daemon restarts and can be managed through the xref:daemon-dashboard.adoc[Dashboard] or the xref:rest-api.adoc[REST API].

== Connection Fields

The following fields are stored for each saved connection.

[cols="1,1,3",options="header"]
|===
|Field |Type |Description

|`id`
|UUID
|Unique identifier, generated automatically when the connection is created.

|`name`
|String
|A human-readable label for the connection, used in the Dashboard picker.

|`driver`
|String
|The JDBC driver type (e.g. `postgres`, `oracle`, `duckdb`, `mysql`).
Derived automatically from the JDBC URL if not supplied explicitly.

|`url`
|String
|The full JDBC connection URL (e.g. `jdbc:postgresql://localhost:5432/mydb`).

|`user`
|String
|The database username.

|`passwordEnv`
|String
|The *name* of the environment variable from which the password is resolved at run time.
No password value is ever stored.

|`createdAt`
|ISO-8601 timestamp
|The date and time the connection was first saved.

|`lastUsedAt`
|ISO-8601 timestamp
|The date and time the connection was most recently used to submit an export job.
`null` if the connection has never been used.

|===

== Password Security

Saved connections never store a password.
Instead, each connection records the *name* of an environment variable (`passwordEnv`).
When an export job runs, the daemon resolves the password from that environment variable using the same mechanism as the command-line interface.

WARNING: Never set `passwordEnv` to a literal password string.
The field must contain the *name* of an environment variable, not its value.

An inline password may be supplied to the ad-hoc *Test* action in the Dashboard.
It is used transiently to verify reachability and is never persisted to the connection registry or written to disk.

For a full discussion of password handling and security constraints, see xref:security.adoc[Security].

== Storing Connections on Disk

The registry is persisted as a JSON array at:

----
~/.sluice/connections.json
----

You can override this path with the `sluice.connections.file` property.
See xref:configuration.adoc[Configuration] for details.

All writes to the file are atomic, so a daemon restart or power interruption will never leave the file in a partially written state.

The following is an example of a single connection entry as it appears on disk.
Note that there is no `password` field — only `passwordEnv`.

[source,json]
----
[
  {
    "id": "a3f1c2d4-e5b6-7890-abcd-ef1234567890",
    "name": "Analytics DB",
    "driver": "postgres",
    "url": "jdbc:postgresql://db.example.com:5432/analytics",
    "user": "analyst",
    "passwordEnv": "ANALYTICS_PW",
    "createdAt": "2025-03-01T09:00:00Z",
    "lastUsedAt": "2025-06-10T14:23:45Z"
  }
]
----

NOTE: Do not edit this file while the daemon is running.
Use the Dashboard or REST API to manage connections so that changes are applied safely.

== Environment Variable Workflow

The recommended workflow is to set the database password as an environment variable before starting the daemon, then reference the variable name when saving a connection.

.Step 1 — export the password in the shell that will run the daemon
[source,bash]
----
export ANALYTICS_PW=secret
----

.Step 2 — save a connection with passwordEnv pointing to that variable name
When you create the connection (via the Dashboard form or the REST API), set the `passwordEnv` field to `ANALYTICS_PW` — not to the password itself.

.Step 3 — the daemon resolves the password at run time
Each time an export job uses this connection, the daemon reads `System.getenv("ANALYTICS_PW")` to obtain the password.
The password is not logged, stored, or included in job records.

TIP: If the environment variable is not set when a job runs, the export will fail with a clear error message indicating the missing variable.
This is preferable to storing a stale or incorrect password on disk.

== Testing Reachability

The Dashboard provides a *Test* action for each saved connection.
When triggered, the daemon attempts to open a real JDBC connection using the same password resolution and connection factory used by export jobs.

The result is reported as one of:

* *Reachable* — the connection was opened and closed successfully.
* *Unreachable* — the connection attempt failed; the error message is displayed.
* *Untested* — the connection has not been tested since the daemon started.

Test results are held in memory only.
They are not written to `connections.json` and are reset when the daemon restarts.

For the *Test* action only, you may supply an inline password in the Dashboard form.
This allows you to verify a connection before the corresponding environment variable has been set in the daemon's environment.
The inline password is discarded immediately after the test and is never persisted.

== Using Saved Connections in Exports

On the Dashboard's export form, a *Use saved connection* picker lists all registered connections by name.
Selecting a connection pre-fills the URL, username, and password-env fields of the form.

When you submit an export job using a saved connection, the daemon updates the connection's `lastUsedAt` timestamp.
This lets you see, at a glance, which connections are actively used.

See xref:daemon-dashboard.adoc[Daemon Dashboard] for a full walkthrough of the export form.
