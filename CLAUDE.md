# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Quarkus + Picocli CLI (`jdbc-export`) that connects to a JDBC database, runs a read-only SQL query, and streams rows to JSON, NDJSON, CSV, TSV, or Parquet. A `daemon` subcommand serves an HTMX dashboard for submitting and monitoring export jobs. Java 25 (auto-downloaded via Gradle toolchain), Quarkus 3.24.x.

## Commands

Use `gradle21w` (on the PATH) instead of `./gradlew`:

```bash
gradle21w build                                                          # build (produces Quarkus runner jar)
gradle21w test                                                           # all tests
gradle21w test --tests "com.example.jdbcexport.jdbc.PasswordResolverTest"  # single test class
gradle21w test --tests "*DuckDbExportIntegrationTest"                    # pattern match

java -jar build/quarkus-app/quarkus-run.jar --help                       # run the CLI
```

`PostgresExportIntegrationTest` uses Testcontainers and requires Docker running. `DuckDbExportIntegrationTest` runs in-process with no external dependencies — prefer DuckDB for new integration coverage unless the behaviour is Postgres-specific.

## Architecture

Everything lives under `com.example.jdbcexport`. The execution pipeline is:

`JdbcExportApplication` (Quarkus main) → `cli/JdbcExportCommand` (`@TopCommand`, all Picocli options, validation, orchestration) → resolves inputs (`jdbc/PasswordResolver`, `jdbc/SqlInputResolver`, `jdbc/SqlSafetyValidator`) → `jdbc/JdbcConnectionFactory` → `jdbc/ResultSetSchemaReader` (reads column metadata up front as `List<ResultSetColumn>`) → `writer/RowWriterFactory` picks a writer → `jdbc/JdbcExporter` streams the forward-only, read-only `ResultSet` row by row into the writer.

Key contracts and conventions:

- **`writer/RowWriter`** is the central interface: `start(ResultSetMetaData)` / `writeRow(ResultSet)` / `finish()` / `close()`. One implementation per format; CSV and TSV share `DelimitedTextRowWriter`. All writers stream — never buffer full result sets. The JSON/NDJSON/delimited writers convert each column via the shared `jdbc/JdbcValueReader.readAsObject` (one `switch` over `java.sql.Types` — booleans/numerics/temporals/binary-as-Base64), so per-type handling lives in one place; Parquet uses the Avro path instead.
- **Metadata sidecar**: `--metadata <path>` writes an `metadata/ExportMetadata` record (tool, format, URL, SQL source, row count, timing, columns) as JSON via `metadata/ExportMetadataWriter`. It is written by `JdbcExportCommand` after the export completes; `--describe` is separate (schema only, no rows).
- **Outbound transforms** (`transform/`) are an *opt-in* ETL layer between the `ResultSet` and the writer (epic issue #5, child issues #6–#15). With no `--transform`/`--transforms-file` the export is the byte-identical fast path (`writeRow(ResultSet)`); a non-empty `TransformPipeline` makes `JdbcExporter` materialise each row into a `Row` (canonical values — the same `JdbcValueReader.readAsObject` shape JSON uses, captured by `jdbc/ValueKind`), run the ordered `OutboundTransformer`s, and call `writer.writeRow(Row)`. Each `OutboundTransformer` has `name()`, `capabilities()` (`TransformCapabilities`), `transformSchema` (resolved once up front so column-reference errors fail fast), and `transform(TransformInput, TransformContext)` returning a sealed `TransformResult` (`Keep`/`Drop`/`Fail`) — explicit rather than null/throw conventions. `TransformPipeline.transform(Row)` stays the stable boundary (exporter/daemon/harness use it); the result machinery is internal. Built-ins (`transform/builtin/`): rename, drop, keep, addStatic, default, map, mask (single/multi-column, marks `SensitiveColumns`), template, and **expression** (sandboxed Apache Commons JEXL — `JexlFeatures` disables method calls/`new`/loops/lambdas/side-effects + `JexlPermissions.RESTRICTED`; compiled once, no value leakage). `TransformRegistry` merges built-ins with external `TransformProvider` SPI impls via `ServiceLoader` (type collision fails fast). `TransformConfigLoader.loadConfig` parses inline shorthand + the JSON `--transforms-file` into a `TransformConfig` (specs + `ErrorStrategy` + `OutputContract`). **Error strategy** (`--on-transform-error` / file `errorStrategy`): fail (default) / skipRow / keepOriginal (snapshots the input row). Explicit transform `name`s must be unique. **Output contract** (`validateOutput`) checks required/unknown fields against the resolved schema before any rows. `TransformMetrics` records counts/timings only (never row data); `transform/metrics/TransformMetricsPublisher` emits Micrometer `sql_transformer_transform_*` meters (config `transform.metrics.*`, slow-transform threshold) to the global registry. Errors use `ExitCodes.TRANSFORM_ERROR` (7). **Caveat**: in transform mode Parquet stores dates/times/timestamps/binary as strings (canonical form) — `AvroSchemaFactory.buildCanonicalSchema` / `AvroValueMapper.mapCanonical`; the fast path keeps native Parquet logical types.
- **Transforms in the daemon + dashboard** (#15): `ExportJobRequest` carries `transforms` (inline specs) + `errorStrategy`; `ExportJobService.run` builds the pipeline, runs the transform-aware export, and captures a `TransformMetrics.Snapshot` onto the `ExportJob`. `DashboardApiResource` `GET /api/transformations` serves `TransformationView` (per-step counts/timings, slow flags, no row values) for the Sluice **Transformations** side-rail view (`sluice/views-transforms.js`); the Dashboard submit form has a transforms textarea + error-strategy selector.
- **Testing transforms**: `transform/testing/TransformTestHarness` (fluent, ships in main for plugin authors) and `TransformFixtureRunner` (JSON input→expected fixtures). JMH benchmarks live in `src/jmh/java` (`me.champeau.jmh` plugin, `gradle21w jmh`) covering the no-transform baseline vs common pipelines over narrow/wide rows.
- **`SqlSafetyValidator`** enforces read-only SQL (`SELECT`/`WITH` only). This is a deliberate product constraint, not incidental.
- **Errors**: throw `error/ExportException` carrying a code from `error/ExitCodes` (1=args, 2=SQL input, 3=database, 4=output write, 5=schema, 6=unsupported format, 99=unexpected). `JdbcExportCommand.call()` is the single place that catches and maps to the process exit code.
- **Parquet** goes through Avro (`parquet-avro`): `parquet/AvroSchemaFactory` builds the schema from JDBC columns, `AvroFieldNameSanitizer` rewrites column names into valid Avro identifiers, and duplicate sanitised names fail fast (users must alias in SQL). Hadoop is a dependency only because parquet-avro requires it; `LocalParquetOutputFile` avoids Hadoop filesystems.
- **`cli/ExportOptions`** is a record snapshot of the parsed options, passed to the factory/writers so they never see Picocli types.
- **Daemon mode** lives in `daemon/`: `DaemonCommand` (Picocli subcommand, blocks on `Quarkus.waitForExit()`), `ExportJobService` (in-memory job registry + virtual-thread executor reusing the CLI pipeline; also exposes `DaemonMetrics` for JVM heap/uptime/counts), `DashboardResource` (serves the single Qute `index` shell only), and `DashboardApiResource` (the JSON API at `/api/*`: jobs, job detail, submit, describe, metrics). The front-end is the **Sluice** ops console — a client-rendered SPA (Kinetic Green theme) under `src/main/resources/META-INF/resources/sluice/` (CSS `rf.css`/`rf-desktop.css`/`ops.css`, JS modules + `app.js` router/poller). It has four side-rail views, all wired live to `/api`: **Dashboard**, **Live Run**, **Schedules**, and **Connections**. Live run metrics are real: `JdbcExporter` takes a `ProgressListener`, and `ExportJob` carries live `rowCount`, output bytes (`Files.size`), fetch size, column count, server info, driver, codec, elapsed and throughput. HTTP lifecycle is decided in `JdbcExportApplication.configureHttp()` *before* `Quarkus.run()`: a real CLI export (args present, not `daemon`) sets `quarkus.http.host-enabled=false` so it never binds a port; `daemon --port/--host` are mapped to system properties there because the server starts before Picocli parses. No-args is left enabled — `JdbcExportApplication.run()` checks `LaunchMode.current() == DEVELOPMENT` (reliable post-boot) and rewrites empty args to `daemon`, so `gradle21w quarkusDev` serves the dashboard with live reload; the packaged jar with no args is a usage error that exits immediately. `--url`/`--user` are validated manually (not `required = true`) because Picocli would otherwise demand them for the `daemon` subcommand too.
- **Saved connections** are operator-managed and **wired**: `ConnectionStore` persists `SavedConnection`s as a JSON array (default `~/.sluice/connections.json`, overridable via `sluice.connections.file`); `ConnectionApiResource` (`/api/connections`) does list/create/update/delete/test. **Passwords are never stored** — only a `passwordEnv` reference resolved at run time by `PasswordResolver`; an inline password may be passed to the ad-hoc test endpoint but is used transiently. Reachability (the "test" result) is kept in memory only. The Dashboard form has a "use saved connection" picker; submitting with one marks it used.
- **Schedules** are operator-managed and **wired**: `Schedule` + `ScheduleStore` persist to a JSON array (default `~/.sluice/schedule.json`, overridable via `sluice.schedule.file`); `ScheduleApiResource` (`/api/schedules`) does list/create/update/delete/run/toggle. A schedule references a `SavedConnection` by id (no credentials in `schedule.json`). `ScheduleTimes` computes next-fire for cron (5-field UNIX via `com.cronutils:cron-utils`), interval, and once triggers, and renders `outputPattern` placeholders (`{date}`/`{time}`/`{datetime}`/`{ts}`). `ScheduleRunner` is a single-thread executor that ticks ~30s and fires due schedules through `ExportJobService`; it starts only in the daemon — gated on `quarkus.http.host-enabled` and skipped under `LaunchMode.TEST` (so tests drive it via `runNow`/`tick`).
- **Dashboard security** (no auth by design — local operator tool): `CsrfFilter` requires the `X-Requested-By: jdbc-export` header on all POST/PUT/PATCH/DELETE; the SPA's `api.js` sets it on every `fetch`. `main()` refuses a non-loopback bind host unless `--allow-remote` is passed (`isLoopbackHost`). When changing the dashboard, keep state-changing endpoints behind the header and don't loosen the bind guard. The SPA renders DB-derived values (output paths, column names, errors, server strings) through `esc()`/`escAttr()` to avoid HTML injection.
- **Plain (non-Quarkus) tests that use `DriverManager` with DuckDB** must register the driver explicitly (see `DuckDbExportIntegrationTest`): a `@QuarkusTest` in the same suite can trigger `DriverManager`'s one-time driver scan under the Quarkus classloader, leaving DuckDB invisible to the test classloader.
- **`JdbcExportCommand` must stay `@Dependent` scoped.** With `@ApplicationScoped`, Picocli binds option values to the CDI client proxy's fields while `call()` executes on the underlying bean with null fields — every option silently fails to bind. `JdbcExportCommandMainTest` (a `@QuarkusMainTest`) guards this; it needs the `--add-opens java.base/java.lang=ALL-UNNAMED` test JVM arg already set in `build.gradle.kts`.
- Postgres and Oracle JDBC drivers are `runtimeOnly`; DuckDB is test-only.
